# Техническое задание: «Голосовой Квиз»

## 1. Назначение проекта

«Голосовой Квиз» — SmartApp Canvas-приложение для Салюта/Sber с голосовым управлением. Пользователь проходит викторину на общую эрудицию, выбирая варианты ответа голосом или кнопками на экране.

Главная особенность проекта — надежное голосовое взаимодействие: приложение должно распознавать команды пользователя, отправлять действия из SmartApp Code в Canvas, озвучивать вопросы, результаты ответов и служебные команды.

## 2. Игровая логика

### 2.1. Формат игры

- В банке вопросов хранится 20 вопросов на русском языке.
- При каждом новом запуске сайта создается новая игровая сессия.
- В одну сессию случайно выбираются 10 вопросов из банка 20.
- Внутри одной сессии вопросы не должны повторяться.
- Порядок вопросов в сессии случайный.
- При перезагрузке страницы или запуске новой игры создается новая сессия.

### 2.2. Вопросы и ответы

Каждый вопрос содержит:

- уникальный `id`;
- текст вопроса;
- 4 варианта ответа;
- ключ правильного варианта.

Варианты ответа обозначаются числами:

- `1`;
- `2`;
- `3`;
- `4`.

Пользователь может выбрать ответ:

- голосом;
- нажатием кнопки на экране.

### 2.3. Обработка ответа

После выбора ответа приложение:

- проверяет корректность выбранного варианта;
- увеличивает счет при правильном ответе;
- подсвечивает выбранный вариант;
- при ошибке показывает правильный вариант подсветкой;
- озвучивает результат ответа;
- если вопрос не финальный, после короткой фазы обратной связи переводит игру к следующему вопросу и озвучивает его отдельным `QUESTION_PROMPT`.

Для согласованности состояния Canvas, `item_selector` и подсказок следующий вопрос не включается в `ANSWER_RESULT`. `ANSWER_RESULT` озвучивает только результат ответа, затем Canvas переводит игру из `feedback` в `question` и отправляет отдельный `QUESTION_PROMPT`.

### 2.4. Завершение игры

Игра завершается после 10 вопросов.

В конце пользователь видит итоговый экран:

- количество правильных ответов;
- общее количество вопросов;
- кнопку «Сыграть ещё».

Голосовая фраза финала:

```text
Вы ответили правильно на X из Y вопросов.
```

## 3. Голосовое управление

### 3.1. Поддерживаемые команды выбора ответа

Для варианта `1`:

- `1`;
- `один`;
- `первый`;
- `номер 1`;
- `номер один`;
- `вариант 1`;
- `вариант один`;
- `первый вариант`;
- `ответ 1`;
- `ответ один`.

Для варианта `2`:

- `2`;
- `два`;
- `второй`;
- `номер 2`;
- `номер два`;
- `вариант 2`;
- `вариант два`;
- `второй вариант`;
- `ответ 2`;
- `ответ два`.

Для варианта `3`:

- `3`;
- `три`;
- `третий`;
- `номер 3`;
- `номер три`;
- `вариант 3`;
- `вариант три`;
- `третий вариант`;
- `ответ 3`;
- `ответ три`.

Для варианта `4`:

- `4`;
- `четыре`;
- `четвертый`;
- `четвёртый`;
- `номер 4`;
- `номер четыре`;
- `вариант 4`;
- `вариант четыре`;
- `четвертый вариант`;
- `четвёртый вариант`;
- `ответ 4`;
- `ответ четыре`.

### 3.2. Служебные голосовые команды

Поддерживаются команды:

- `повтори вопрос` — повторить текущий вопрос или финальный результат, если игра завершена;
- `мой счёт` / `мой счет` — озвучить текущий счет;
- `начать заново` — начать новую игру;
- `новая игра` — начать новую игру;
- `сыграть еще` / `сыграть ещё` — начать новую игру.

### 3.3. Принцип голосового взаимодействия

SmartApp Code отвечает за:

- распознавание фраз пользователя;
- преобразование фраз в action;
- отправку action в Canvas через `smart_app_data`;
- обработку событий, отправленных Canvas через `sendData`;
- вызов `$reactions.answer` для озвучки.

Canvas отвечает за:

- игровое состояние;
- применение действия;
- формирование voice event;
- отправку voice event в SmartApp Code;
- управление очередью озвучки.

- `src/game/states.js` — игровые фазы.
- `src/game/config.js` — тайминги и лимиты.

### 4.3. SmartApp Code

Ключевые файлы:

- `scenario/src/sc/gameEvents.sc` — обработка событий из Canvas.
- `scenario/src/js/actions.js` — отправка действий в Canvas.
- `scenario/src/js/contract.js` — id action/event.
- `scenario/src/js/getters.js` — чтение state и event payload.
- `scenario/src/js/reply.js` — формирование raw replies и suggestions.
- `scenario/src/js/voicePhrases.js` — fallback-фразы сценария.

## 5. Игровые состояния

Используются фазы:

- `question` — ожидается ответ пользователя;
- `feedback` — ответ принят, идет краткая фаза обратной связи перед переходом;
- `result` — игра завершена.

В состоянии игры хранятся:

- `gameSessionId`;
- `questions`;
- `currentQuestionIndex`;
- `score`;
- `selectedOption`;
- `revealedCorrectOption`;
- `phase`.

## 6. Контракт Canvas и SmartApp Code

### 6.1. Входящие action из SmartApp Code в Canvas

Поддерживаются action:

- `select_option`;
- `repeat_question`;
- `current_score`;
- `restart_game`.

Action `select_option` должен содержать:

```json
{
  "type": "select_option",
  "option": "1"
}
```

Допустимые значения `option`:

- `"1"`;
- `"2"`;
- `"3"`;
- `"4"`.

### 6.2. Исходящие event из Canvas в SmartApp Code

Поддерживаются event:

- `game_started`;
- `question_prompt`;
- `answer_result`;
- `score_report`;
- `result_prompt`;
- `invalid_action`.

Каждый event должен содержать `payload.phrase`.

Дополнительно payload может содержать:

- `suggestions`;
- `phase`;
- `gameSessionId`;
- `questionIndex`;
- `score`;
- `selectedOption`;
- `correctOption`;
- `voiceTurnId`.

### 6.3. Защита контракта

Canvas валидирует:

- корректность входящего action;
- допустимость варианта ответа;
- наличие непустой фразы в исходящем event;
- тип `suggestions`;
- тип `voiceTurnId`;
- тип `phase`.

Некорректные действия и события игнорируются.

## 7. Voice Scheduler

### 7.1. Назначение

`VoiceEventScheduler` управляет отправкой voice events из Canvas в SmartApp Code.

Он решает задачи:

- защита от потери фраз;
- защита от устаревших callback;
- прерывание текущей озвучки при новом пользовательском действии;
- дедупликация очереди;
- отбрасывание устаревших событий;
- диагностирование жизненного цикла voice events.

### 7.2. Политики отправки

Используются политики:

- `ENQUEUE` — поставить событие в очередь;
- `REPLACE` — очистить pending-очередь и поставить новое событие;
- `INTERRUPT` — прервать текущую озвучку, очистить очередь и отправить новое событие.

Критические пользовательские действия используют `INTERRUPT`:

- выбор ответа;
- новая игра;
- повтор вопроса;
- мой счет;
- запрос результата.

### 7.3. `voiceTurnId`

Каждой отправке присваивается `voiceTurnId`.

Если старая озвучка была прервана, а ее callback вернулся позже, scheduler игнорирует этот callback. Это защищает новую озвучку от повреждения старым асинхронным ответом.

### 7.4. Диагностика

Диагностический режим включается переменной:

```dotenv
REACT_APP_VOICE_DEBUG="true"
```

В этом режиме scheduler пишет события:

- `queued`;
- `sent`;
- `settled`;
- `interrupted`;
- `dropped_stale`;
- `ignored_stale_callback`;
- `ignored_stale_settle`;
- `send_error`.

## 8. UI

На экране отображаются:

- название игры;
- текущий счет;
- кнопка «Новая игра»;
- номер текущего вопроса;
- текст вопроса;
- 4 кнопки ответа;
- финальный экран после завершения игры.

Нижняя часть экрана после блока вопроса и ответов должна оставаться чистой: без подсказок, toast-сообщений и дополнительного текста.

Обратная связь по правильности ответа сейчас реализована через подсветку кнопок и озвучку. Текстовая обратная связь может быть возвращена позднее отдельным решением.

## 9. Тестирование

В проекте есть unit-тесты для:

- игрового движка;
- voice contract;
- voice queue;
- voice scheduler.

Проверяются:

- создание новой игровой сессии;
- выбор 10 вопросов из 20;
- отсутствие повторов в сессии;
- обработка правильного и неправильного ответа;
- разделение результата ответа и следующего вопроса на `ANSWER_RESULT` и `QUESTION_PROMPT`;
- валидация action/event;
- защита от устаревших callback;
- прерывание текущей озвучки;
- stale event filtering.

Команда запуска:

```bash
npx react-scripts test --watchAll=false
```

```bash
npm run build
```

## 11. Потенциальные уязвимости и риски системы

### 11.1. Зависимость от поведения Sber/Salute TTS

Canvas может управлять очередью и отправкой событий, но не гарантирует физическое мгновенное прерывание уже начатой озвучки на стороне платформы. Даже при `INTERRUPT` старый TTS может продолжить звучать, если runtime Салюта не остановит его сразу.

### 11.2. `sendData` callback не равен окончанию речи

Callback `assistant.sendData` подтверждает обработку события, но может не означать, что TTS полностью завершился. Поэтому часть синхронизации UI и голоса остается эмпирической.

### 11.3. Тайминги остаются частично эвристическими

Параметры `VOICE_ACK_TIMEOUT_MS`, `VOICE_BLOCKING_MIN_SETTLE_MS`, `NEXT_QUESTION_DELAY_MS` подобраны практически. На разных устройствах и версиях runtime возможны отличия в скорости озвучки.

### 11.4. SmartApp Code тестируется слабее, чем React-часть

Unit-тесты покрывают React/game/voice-модули, но не проверяют полноценно `.sc` сценарии в SmartApp Studio. Возможны расхождения между локальной логикой и реальным распознаванием на платформе.

### 11.5. Распознавание чисел может ошибаться

Числа `1`, `2`, `3`, `4` должны распознаваться стабильнее английских букв, но ASR может путать короткие однословные ответы, особенно при шуме, обрыве речи или быстром ответе во время TTS.

### 11.6. Агрессивное прерывание может реагировать на ложные команды

Критические пользовательские действия используют `INTERRUPT`. Если платформа ошибочно распознает фразу как команду, текущая озвучка будет прервана.

### 11.7. Нет серверной валидации

Вся логика находится на клиенте. Для текущего quiz-MVP это допустимо, но приложение не защищено от ручного изменения состояния в браузере.

### 11.8. Нет персистентности прогресса

Сессия намеренно сбрасывается при каждом запуске сайта. Это соответствует текущему требованию, но пользователь потеряет прогресс при перезагрузке страницы.

### 11.9. Возможны рассинхроны UI и TTS

Переход от результата ответа к следующему вопросу выполняется по таймеру. Scheduler удерживает blocking-события минимум заданное время, но callback `sendData` не гарантирует физическое окончание речи, поэтому на разных устройствах возможны небольшие отличия в моменте начала следующей реплики.

### 11.10. Архив `scenario.zip` может устареть

Если изменять `scenario/`, но не пересобирать `scenario.zip`, в SmartApp Studio может быть загружена устаревшая версия сценария.

### 11.11. Fallback зависит от корректного state payload

Fallback-сценарий определяет текущую фазу через state приложения. Если платформа передаст неполный или устаревший state, fallback может дать неидеальную подсказку.

### 11.12. Нет end-to-end тестов на устройстве

Пока нет автоматической проверки полного пути:

```text
голос пользователя -> SmartApp Code -> Canvas action -> game state -> sendData -> SmartApp Code event -> TTS
```

Эта цепочка должна регулярно проверяться вручную в SmartApp Debugger и на целевых устройствах.
